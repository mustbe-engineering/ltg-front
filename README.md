# LTG Front

This is the frontend application for the LTG project, built with **Vue 3**, **TypeScript**, and **Vite**.

It follows a **Vertical Slicing Architecture** to keep features self-contained and scalable.

## 🚀 Tech Stack & Versions

This project relies on the latest stable versions of its core dependencies:

*   **Vue.js:** `^3.5.x`
*   **Vite:** `^6.x` (or latest)
*   **TypeScript:** `^5.x`
*   **Tailwind CSS:** `^4.x` (Modern styling engine)
*   **Node.js:** Recommended `v20` (LTS) or `v22`.
*   **Package Manager:** `npm` (v10+)

## 🏗 Architecture

### Vertical Slicing
Instead of grouping by technical layers (components/views/stores), we group by **Feature**:

```
src/
  features/
    blog/           # Blog feature
      components/   # Components only used in Blog
      views/        # Pages/Views for Blog
      api/          # API calls specific to Blog
    home/           # Home feature
    shared/         # Shared components/utils
  services/         # Global services (e.g. CMS Client)
  router/           # Application Routing
```

### CMS Integration (Hugo)
This application is designed to work with a **Headless Hugo CMS**.

- **Production:** Fetches content from static JSON files generated by Hugo (e.g., `/api/posts.json`).
- **Development:** Uses a **Mock Service** (`src/services/contentService.ts`) to simulate the CMS response structure without needing the full Hugo backend running.

## 🚀 Getting Started

### Prerequisites
- Node.js (v18+ recommended)
- npm

### Setup

1. **Clone the repository:**
   ```bash
   git clone <repo-url>
   cd ltg-front
   ```

2. **Install dependencies:**
   ```bash
   npm install
   ```

3. **Environment Setup:**
   Copy the example environment file:
   ```bash
   cp .env.example .env
   ```

4. **Run Development Server:**
   ```bash
   npm run dev
   ```
   Access at `http://localhost:5173`

## 🛠 Build & Deploy

To build for production:

```bash
npm run build
```

The output will be in `dist/`. Ensure your deployment pipeline copies the Hugo-generated JSONs into `dist/api/` or configures the fetch calls to point to the CMS origin.

## 🧪 Mocking Data & CMS Structure

This project relies on a specific JSON structure to function correctly with the CMS.

📄 **[See MOCK_DATA_STRUCTURE.md](docs/MOCK_DATA_STRUCTURE.md)** for the full schema contract between Frontend and CMS.

To add new mock data for development, edit `src/services/contentService.ts` and add entries matching those schemas.
